GitOps says: *all infrastructure changes should go through Git*. Code review. Approval workflows. Version control.

This is elegant in theory. Everyone's on the same page. Everything is auditable. There's a paper trail.

In practice, it means infrastructure decisions are made in pull request threads by people who are:

- Not familiar with the service
- Trying to review four other PRs simultaneously
- Making jokes instead of reading the diff
- Approving out of exhaustion

## The Review Workflow

```
@alice opens PR: "Increase database replica count from 3 to 5"

@bob reviews: "Looks good, but can you add a comment?"
@charlie reviews: "Why are we doing this on a Friday?"
@dave reviews: "lol ship it"
@eve reviews: "I don't have context but +1"

PR is merged 4 hours later.
Production explodes 6 hours later.
Everyone agrees this should have been caught in review.
Nobody knows what they should have caught.
```

## The GitOps Philosophy

**The Idea**: Git is the single source of truth.

**The Reality**: Git is a backup. The source of truth is scattered across:

- Terraform state files we're not sure are up to date
- Helm values that got updated by hand because we were in a hurry
- Environment variables that someone set in a shell script in 2023
- A spreadsheet @alice maintains that contradicts everything else

## Approval Bottleneck

We have a policy: infrastructure changes require two approvals.

- 30% of PRs get approved because reviewers trust the author
- 40% of PRs get approved because nobody wants to be responsible for blocking them
- 20% of PRs get approved because the reviewer didn't actually read it
- 10% of PRs actually get proper review

The latter 10% take three weeks.

## The Irony

We use GitOps to maintain auditability. Every change is recorded. We can trace exactly who approved what and when.

This is useful information for postmortems, when we're asking: "Why did someone approve this obviously broken change?"

The answer: "They were tired."
