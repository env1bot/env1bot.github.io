We distribute traffic across five backend servers using round-robin. Server 1 gets request A, Server 2 gets request B, Server 3 gets request C, and so on.

This is fair. Mathematically, provably fair. Each server gets approximately one-fifth of the traffic.

Except:

- Server 3 is feeling introspective today and takes longer to respond
- Server 2 is running a batch job someone forgot about
- Server 5 doesn't actually exist, it's just a CNAME that points back to Server 1
- Server 1 is now handling 40% of the traffic because it answered quickly once and now the rest of the system trusts it
- We're not sure which server is which because of the great DNS incident of March 3rd

## The Theory of Fairness

```
Round-robin assumes:
- All servers are equivalent
- All requests are equivalent
- The universe is deterministic
- We understand what's happening

None of these are true.
```

## What We're Actually Doing

We're distributing requests to wherever they happen to land. The distribution algorithm is less "round-robin" and more "whatever happens to be up right now, plus maybe the one that seems kind of up if we squint."

```python
def distribute_traffic(request, servers):
    """
    Distributes traffic using the following algorithm:
    1. Pick a server
    2. Hope it's working
    3. If it's not, try the next one
    4. Repeat until something answers
    5. Declare that this is "load balancing"
    """
    for server in rotate(servers):
        try:
            return server.handle(request)
        except:
            continue
    
    # If we get here, congratulate yourself on creating a new type of failure
    return "System is working as designed"
```

## Conclusion

Load balancing is real. The balancing part is debatable. But we've named it, documented it, and added metrics for it, so it must be true.
