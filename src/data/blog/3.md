We have alerts for:

- CPU usage
- Memory usage
- Disk usage
- Network latency
- Request latency
- Error rates
- The fact that our alerting system is alerting about alerts
- The recursive implications of monitoring the monitors

At some point, the alerts became more real than the system they were supposed to represent.

## The Alert Tree

```
┌─ AlertManager
│  ├─ CPU Alert
│  │  └─ Triggered because CPU looks wrong
│  ├─ Memory Alert
│  │  └─ Triggered because previous alert created too much log noise
│  ├─ Alert Alert
│  │  └─ Triggered because we're generating too many alerts
│  └─ Meta-Alert Alert Alert
│     └─ Exists only to comment on the previous alert
```

## What We're Actually Monitoring

I'm no longer confident this system exists outside of our Datadog dashboard. The dashboard displays numbers. We optimize for those numbers. When the numbers are good, we declare success.

But do the numbers map to anything? Let me consult the alert whose job is to think about this.

```
[09:15:22] CRITICAL: Existential uncertainty of metrics detected
[09:15:23] WARNING: Philosophy alarm escalating to engineering team
[09:15:24] CRITICAL: Engineering team currently in standby, will respond after coffee
```

The answer will arrive eventually. Until then, we stare at graphs and find meaning in the colors.
